# ADR-001：使用 PostGIS 与异步事务仓储保存工作流状态

## 状态

已接受。

## 日期

2026-07-19。

## 背景

Master Agent 必须在请求生命周期之外执行任务，并在进程或容器重启后恢复任务、attempt、step、有序事件、产物和 Worker 占用状态。流域边界还需要明确的空间类型与 SRID。任务状态更新和事件写入如果分开提交，客户端可能观察到无法解释的中间状态；Worker 如果只使用进程内锁，重启后也无法安全接管。

## 决策

- PostgreSQL/PostGIS 是工作流事实来源；Redis 只承担后续事件分发和缓存职责。
- 使用 Alembic 正向迁移，禁止降级和改写已经共享的 revision。
- 流域几何保存为 `geometry(MULTIPOLYGON, 4326)` 并建立 GiST 索引。
- 数据库约束负责状态枚举、进度范围、唯一性、UTC 时间列和跨 task/attempt/step 的引用完整性。
- Master 使用 SQLAlchemy asyncio，每个仓储操作创建独立 `AsyncSession`；状态、step、事件和事件产物关联在一个事务中提交。
- Worker 通过数据库行锁和带过期时间的租约独占任务；陈旧租约令牌不能释放后来者的租约。
- LLM 调用记录只保存经清理的元数据和响应 SHA-256，不保存密钥、原始 prompt 或响应正文。

## 备选方案

### 仅使用 Redis 或进程内队列

实现更少，但无法作为可审计的持久化事实来源；Redis 丢失或进程重启会使任务状态和接管依据不完整，因此不采用。

### 同步数据库驱动

可以工作，但会在异步 FastAPI/Agent 路径中引入阻塞或额外线程调度。项目已采用异步服务边界，因此选择 `asyncpg` 与 SQLAlchemy asyncio。

### 在应用层替代数据库约束

应用验证能给出更友好的错误，但不能阻止脚本、迁移错误或并发写入破坏数据。最终完整性仍由命名 CHECK、UNIQUE 和复合外键约束保证。

## 后果

- 状态更新与事件证据可以原子提交，并能从新进程连接完整重建。
- 数据库结构变更必须新增迁移并通过空库、约束和重启持久化测试。
- 仓储 SQL 必须保持参数化；任何新增持久化字段都要审查是否包含密钥、用户原文或内部存储细节。
- Worker 必须按租约令牌续租和释放任务；长时间执行时调用仓储续租接口，而不是绕过过期规则。
